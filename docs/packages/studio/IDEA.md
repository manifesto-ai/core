## 1. 핵심 비전

> **AI Agent가 실행할 수 있는 비즈니스 로직을 '검증 가능한 구조'로 만든다**

Studio는 Manifesto 기반 도메인을 시각적으로 편집하고, 실시간으로 검증하며, 시뮬레이션 테스트할 수 있는 웹 IDE다.

### 1.1 왜 Studio인가?

Manifesto Core와 Compiler가 강력한 기반을 제공하지만, 개발자들이 실제로 사용하기까지의 장벽이 높다:

- SemanticPath, Expression DSL, Effect System 등 **개념 밀도가 높음**
- DAG 기반 의존성 흐름을 **머릿속으로 추적하기 어려움**
- 변경의 영향 범위를 **즉시 파악하기 어려움**
- Policy/Precondition이 왜 막히는지 **디버깅이 어려움**

Studio는 이 모든 것을 **시각적, 상호작용적, 실시간**으로 해결한다.

## 2. Wow Points (차별화 포인트)

### 2.1 Explain - "왜 이 값인지"

```
[노드 선택] → [Explain 패널]
├── 현재 값: 25000
├── 계산 과정: cart.subtotal + shipping.fee
├── 의존 경로: data.items → derived.subtotal → derived.total
└── 입력 값 변경 시뮬레이션
```

Runtime의 `explain(path)` API를 시각화하여, 어떤 값이든 "왜 그런지"를 트리 형태로 보여준다.

### 2.2 Impact - "무엇이 영향받는지"

```
[값 변경 의도] → [Impact Preview]
├── 직접 영향: derived.canCheckout
├── 간접 영향: ui.checkout.disabled
└── 영향받는 Action: checkout, updateCart
```

Runtime의 `getImpact(path)` API를 활용해, 수정 전에 영향 범위를 미리 보여준다.

### 2.3 Preconditions/Policy - "왜 비활성인지"

```
[비활성 버튼 클릭] → [Policy Debugger]
├── Action: checkout
├── 충족된 조건: isLoggedIn ✓
├── 미충족 조건: cart.total > 0 ✗ (현재: 0)
└── 추천: cart에 아이템 추가 필요
```

Policy가 왜 막혔는지, 어떤 조건을 충족하면 되는지 명확히 보여준다.

## 3. 타겟 사용자

### 3.1 Primary: AI Agent 개발자

- Manifesto 기반 Agent를 만드는 개발자
- 비즈니스 로직을 안전하게 Agent에게 노출해야 함
- **핵심 니즈**: "Agent가 이해하고 조작할 수 있는 구조를 빠르게 만들고 싶다"

### 3.2 Secondary: 풀스택 개발자

- React/Next.js 앱에 Manifesto를 도입하려는 개발자
- 복잡한 폼 검증, 상태 관리가 필요한 경우
- **핵심 니즈**: "로직 흐름을 시각화하며 개발하고 싶다"

### 3.3 Tertiary: 기획자/PM

- 비즈니스 로직을 개발자와 함께 설계하는 역할
- 요구사항을 자연어로 표현하고 결과를 확인하고 싶음
- **핵심 니즈**: "기획 의도가 코드로 어떻게 변환되는지 보고 싶다"

## 4. 기능 아이디어

### 4.1 입력: 문서형 + 블록 채팅 하이브리드

**문서형 캔버스**
- Notion 스타일의 블록 기반 에디터
- 각 블록이 Schema, Derived, Action 등의 Fragment가 될 수 있음
- 드래그앤드롭으로 구조 재배치

**블록별 채팅**
- 각 블록 옆에 AI 채팅 버튼
- "이 조건을 더 엄격하게" → 해당 블록만 수정 제안
- Fragment 단위 수정 (전체 재생성 아님)

```
┌─────────────────────────────────────┐
│ [Schema Block] 장바구니 아이템      │ [💬]
│   - id: string                      │
│   - quantity: number (min: 1)       │
├─────────────────────────────────────┤
│ [Derived Block] 총합 계산           │ [💬]
│   items.reduce((sum, i) => ...)     │
├─────────────────────────────────────┤
│ [Action Block] 결제하기             │ [💬]
│   precondition: canCheckout         │
└─────────────────────────────────────┘
```

### 4.2 시각화: React Flow DAG

**노드 타입**
- 🔵 Data 노드 (Schema 필드)
- 🟢 Derived 노드 (계산된 값)
- 🟡 State 노드 (UI 상태)
- 🔴 Action 노드 (실행 가능한 행동)

**인터랙션**
- 노드 호버: 현재 값 표시
- 노드 클릭: 상세 패널 (Explain, Impact, Edit)
- 엣지 색상: 의존성 방향과 타입 표현
- 실시간 업데이트: 값 변경 시 영향받는 노드 하이라이트

### 4.3 테스트: 실시간 시뮬레이션

**시나리오 정의**
```
Scenario: 결제 플로우 테스트
Given:
  - data.items: [{id: "A", quantity: 2, price: 10000}]
  - data.user.isLoggedIn: true
When:
  - action: checkout
Then:
  - derived.total: 20000
  - state.checkout.completed: true
```

**시나리오 입력 형식**
- 자연어 + JSON 혼합 (MVP)
- 폼 UI (추후)

**시뮬레이션 실행**
- 스냅샷 비교: 시나리오 전/후 상태
- 스텝 실행: Action을 하나씩 실행하며 상태 변화 관찰
- Effect 모킹: apiCall 등 외부 Effect는 mock response로 대체

### 4.4 검증: 이슈 패널

```
┌──────────────────────────────────┐
│ Issues (3)                       │
├──────────────────────────────────┤
│ ❌ MISSING_DEPENDENCY            │
│    derived.total requires        │
│    undefined path: data.tax      │
│    [Fix: Add tax field]          │
├──────────────────────────────────┤
│ ⚠️ UNUSED_PATH                   │
│    data.legacy is not referenced │
│    [Remove] [Ignore]             │
├──────────────────────────────────┤
│ ℹ️ SUGGESTION                    │
│    Consider adding validation    │
│    for quantity < 0              │
└──────────────────────────────────┘
```

`validateDomain` 이슈를 실시간으로 표시하고, 원클릭 수정 제안.

### 4.5 인사이트: 성능 분석

- 전파 깊이 (propagation depth)
- 핫 패스 (자주 업데이트되는 경로)
- 복잡도 지표 (노드 수, 엣지 수, 최대 팬아웃)

### 4.6 출력: 코드/JSON 익스포트

```typescript
// Generated by @manifesto-ai/studio
export const cartDomain = defineDomain({
  id: 'cart',
  name: 'Shopping Cart',
  dataSchema: z.object({
    items: z.array(itemSchema),
    // ...
  }),
  // ...
});
```

- TypeScript 코드 (defineDomain)
- JSON (도메인 정의)
- Fragment JSON (개별 조각)

## 5. 결정 사항

| 항목 | 결정 | 이유 |
|------|------|------|
| **입력 UX** | 문서형 + 블록 채팅 | 자연어와 구조화 편집의 장점 결합 |
| **기본 LLM** | OpenAI | 안정성과 비용 밸런스, Provider 추상화 적용 |
| **시나리오 입력** | 자연어 + JSON (MVP) | 폼 UI는 오버엔지니어링, 추후 추가 |
| **PoC 순서** | 결정론적 루프 → LLM | "잘 되는 것처럼 보이지만 불안정" 방지 |
| **배포 방식** | 오픈소스 + SaaS | 커뮤니티 검증 + 수익 모델 |
| **협업 기능** | Post-MVP | SaaS 안정화 후 추가 |
| **MVP 위치** | apps/studio | Compiler 검증 목적 |

## 6. 기술 고려사항

### 6.1 Schema Descriptor DSL

**문제**: Zod 스키마는 JS 함수라서 직렬화/역직렬화가 어려움

**해결**: JSON Schema-like DSL 도입
```typescript
// Schema Descriptor (직렬화 가능)
const descriptor = {
  type: 'object',
  properties: {
    quantity: { type: 'number', minimum: 1 }
  }
};

// Runtime에서 Zod로 변환
const zodSchema = descriptorToZod(descriptor);
```

### 6.2 Effect 모킹

**문제**: `apiCall`, `navigate` 등 실제 Effect를 테스트에서 실행할 수 없음

**해결**: Mock Handler Registry
```typescript
const mockHandlers = {
  apiCall: (endpoint, payload) => {
    if (endpoint === '/api/checkout') {
      return { success: true, orderId: 'mock-123' };
    }
  }
};

runtime.runEffect(effect, { handlers: mockHandlers });
```

### 6.3 Compiler 연동

**결정론적 루프 먼저**
1. 수동 Fragment 편집
2. Linker → Verifier 실행
3. 이슈 해결
4. 도메인 완성

**LLM 연동은 이후**
1. 자연어 입력
2. Compiler Pass 실행 → Fragment 생성
3. 결정론적 루프로 검증

## 7. MVP 템플릿

### 7.1 E-commerce Cart
```
- data: items[], user
- derived: subtotal, tax, total, canCheckout
- action: addItem, removeItem, checkout
- policy: checkout requires isLoggedIn && total > 0
```

### 7.2 Form Validation
```
- data: email, password, confirmPassword
- derived: isEmailValid, isPasswordStrong, passwordsMatch
- derived: canSubmit = all validations pass
- action: submit
```

### 7.3 Search + Async
```
- data: query, results[]
- state: isLoading
- action: search (with apiCall effect)
- derived: hasResults, resultCount
```
